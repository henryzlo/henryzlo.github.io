Advantages of  object encapsulation:

 

1.      Clear statement of functionality in use — what is intended/provided by a Bank Account or a Stack.

2.      Partition of responsibility/code.  Important if many programmers.

3.      Can share General Purpose objects across many apps (if do them right). Saves coding. At worst have to make   minor improvements later.

4.      Safety of contents of Stack, etc.  Primitive functions can check argu­ments, etc., whereas in open coding we tend to do the minimum. Sharing the implementation here, so worth it.

5.      Worth it to carefully test implementation code, remove all bugs, put special trace capabilities in, etc.

6.      Debugability.  No mystery changes to data—can breakpoint the functions that change data.

 

Algorithm Analysis: Big Oh Notation, Weiss Chap. 5

 

The basics of big-oh are hopefully review, covered in CS210

 

Many computer algorithms are sized by their input.  For ex, the qual has N input lines, where N = 10 for testing, 100000 for a real run.  If we double the number to 200000, does the time it takes double? or what? 

 

In general, let T(N) stand for the time taken by a program for input of size N.

 

Linear time:  If T(N) is a linear function of N, it has a straight-line graph, and we say T(N) = O(N).

Quadratic time: If T(N) is a quadratic function of N, it has a parabolic graph, steeper and steeper, and we say T(N) = O(N2).

Cubic...

Exponential...  really fast growth

 

Logarithmic...  really slow growth

Example: for log-base-2, since 29 = 512 and 210 = 1024, we see that log(512) = 9 and log(1024) = 10, just a little bit more gained over 512 units.

 

Def, pg. 201 (177):  T(N) is O(F(N)) if there are positive constants c and N0 such that T(N) <= c F(N) when N >= N0.

That is, for big enough N, T(N) is bounded by some constant multiple of F(N).  See def of big omega, etc. Also little oh, but not as important in C.S.

 

Note: N and T(N) are assumed positive here, N measuring problem size and T measuring some resource such as elapsed time, CPU time, or memory.

 

Thus we say 100N is O(N), and even 100N + 200 is O(N).

 

There are other algorithms where programmers have to be made aware of order of growth.  For example, two different algorithms for sorting an array of numbers, one grows like n2 and the other like n.log(n)—very important difference.

 

There is also a "big omega" or simply "omega" notation.  We say the running time T(n) is W(n2) ("big omega" of n squared) if there are constants c and n0 such that for n > n0,  T(n) ³ cn2, so that T(n) grows at least as fast as n2.

 

Note that both for O() and W(), we only care about leading power term in poly­nomial.  3n3 + n2 + 2n + 17 and .001n3 + 170n2 - 88 are both O(n3) and W(n3), because sooner or later the leading cubed term is much bigger than all the other terms.

 

To have a good estimate on the running time of a program, you will need both a "big oh" and a "big omega" estimate.

 

Idea of O(1): bounded by constant, so if T(N) -> 30 as N-> infinity, then we say T(N) = O(1).

 

Rule for Sums:  if T1(N) = O(F(N)) and T2(N) = O(G(N)), then T1 + T2 = O(max(f(N), g(N)), i.e., the biggest contribution to a sum of a few terms dominates the sum.

 

 

Rule for Products: if T1(N) = O(F(N)) and T2(N) = O(G(N)), then T1 * T2 = O(f(N) * g(N)), i.e. the order formulas just multiply like the original functions.

 

Example:  (3n2 + 17)*(4 n2 + 1) = O(n2) * O(n2) = O(n4)

 

             (3n2 + 17)/(4 n2 + 1) = O(n2) / O(n2) = O(1)

 

In a sum, if there are lots of terms of same O(N) size, we need to multiply the number of terms by the size of one.

 

Example: Sum of i from 1 to N:  average size N/2, N terms, so Sum = O(N2).  Exact formula: N(N+1)/2.

 

Loops

 

Weiss, pg. 202 (178) rule: the running time of a loop is at most the running time of a statement in the loop times the number of iterations.

 

Example of Bubblesort (a slow sort)

 

    /* function to sort array of ints in A[0] to A[n-1]                                     */

    int bubblesort(int A[ ], int n)

    {

         int i, j, temp;

 

         for(i = 0; i < n-1; i++)              [n passes of loop]

             for(j = n-1; j > i; j--)          [n-i passes of loop]

                  if (A[j-1] > A[j])    {           /* out of order: swap */   [body is O(1)]

                      temp = A[j-1]; A[j-1] = A[j]; A[j] = temp;

                  }

    }

Work from inside out: body of loop is O(1).  Inner loop does n-i passes of O(1).  Outer loop does n passes i=0, i=1, …i = n-1, or counting inner passes, n, n-1, n-2, … 1, a total of n(n+1)/2 passes of the inner loop, itself of O(1).

Thus T(n) = n(n+1)/2 * O(1) =O(n2 +n) = O(n2).

 

Now there are a number of algorithms that sort in much less than n2 time.  We'll review them later in the term.

 

Recursive Functions: not in Weiss Chap 5, see Chap 7, esp. pg.323-327 (281-285). Only partially covered in CS210.

 

The classic example of recursive function calls. In C or Java (for n not too large, since int overflows at 2G),

 

  int factorial(int n)

  {

    if (n <= 1)

      return 1;

    else

      return n*factorial(n-1);

  }

 

Logic before call to fact(n-1), have O(1) operations, so: T(n) = c + T(n-1).

By same rule, T(n-1) = c + T(n-1-1) = c + T(n-2)

So T(n) = 2c + T(n-2);  Similarly T(n) = 3c + T(n-3) . . . T(n) = (n-1).c + T(1),

but T(1) is O(1), so T(n) = (n-1).c + k = O(n).

 

Also, we find that the iterative method to calculate the factorial also has T(n) = O(n).

But recursion is not always well-behaved!

 

Note that in factorial, a well-behaved recursive method, there is only one recursive call, that’s GOOD.  But another thing to check is how much progress is made going from the current case to the recursive case. We look at the argument to factorial and see n-1, only a little bit of progress (towards smaller, easier problems) over the current case of n, NOT GREAT.

 

So single recursion with f(n-1) can be OK.

 

But when we see multiple recursive calls in the body of the recursive function, that’s a big warning flag, especially if it also makes little progress towards smaller problems as it calls down.

 

Now consider function power2 to calculate 2n (valid for int n ≥ 0):

 

  int power2(int n)

  {

    if (n == 0)

      return 1;

    else

      return power2(n-1) + power2(n-1);

  }

 

T(n) = c + 2T(n-1);  T(n-1) = c + 2T(n-2); . . . ; T(2) = c + 2T(1); T(1) = k.

 

Substituting, T(2) = c + 2k; T(3) = c + 2(c + 2k); T(4) = c + 2(c + 2(c + 2k))

and so on.  Let's ignore everything to do with c, since it only makes a positive contribution.

 

Then T(2) > 2k; T(3) > 4k; T(n) > 2n-1.k.  So T(n) takes time proportional to 2n, (const times 2n-1) a function that grows faster than any polynomial.  This is the exponential growth rate that we try very hard to avoid. This function is unusable in practice for n > 10 say.

 

Luckily it's easy to recode this to multiply by 2 rather than add, or do it iteratively—either way it then looks like the fact function and is similarly T(n) = O(n).

 

The problem in this code is the double recursion without making much progress in making subproblems smaller. Each call involves 2 recursive calls, so the call tree looks like a big binary tree. 

 

However, double recursion is not always so bad, if there’s some splitting of the work too!  In other words, it’s OK if better progress is made going from the current problem to the sub-problems.  F(n/2), say, instead of F(n-1).

 

Considered mergeSort, which has double recursion yet avoids exponential time. Its secret is the fact that it does the recursive call with parameter (n/2), i.e to do the sort of on one half the array.

 

T(n) = T(n/2) + T(n/2) + O(n)  = 2*T(n/2) + O(n)     Doing the two half-size subsorts, then merging their results

 

Can use repeated substitution, as on pg. 345, and find that T(N) = O(NlogN).  This is a good sort, barely above O(N) performance.

 

Note Theorem on pg. 327 for generalized divide and conquer.  A=#recursive calls in body, B=scale-down factor in division of work (B=2 for half and half), k for work of body other than recursive calls, so that it is O(Nk)

 

For merge sort T(n) = 2*T(n/2) + O(n)   

A=B=2, k = 1, Bk = 2, so A = Bk,  T(n) = O(n log n)   Checking with the above

 

For example if T(n) = 2*T(n/2) + O(n2)    (like merge sort but with a more expensive operation than merge)

A=B=2, k = 2, Bk = 4, so A < Bk,  T(n) = O(n2)   Here the recursion contribution is nlogn, below O(n2)
