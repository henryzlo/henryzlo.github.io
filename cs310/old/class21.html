A cycle in a digraph is simply a path that returns to its starting vertex, defined a little after Fig. 14.1.  An acyclic digraph is also called a DAG, short for directed acyclic graph. These graphs show up in lots of apps.  For example, the graph of course prerequisites

CS110 -> CS210 -> CS310

      -> CS240 ->

is a DAG.  A cycle in prereq’s would be ridiculous.

 

Another way of thinking of a DAG is as a partial order.  Not all element pairs have an order, but some do, and the ones that do must be consistent.  So CS110 < CS210 < CS310, and so CS110 < CS310, but CS210 and CS240 have no order between them.

 

Suppose a student took only one course per term in CS.  Then they would be finding a sequence that satisfies the partial order requirements, for example CS110, CS210, CS240, CS310.   Another possible sequence is CS110, CS240, CS210, CS310.  One of these fully ordered sequences that satisfy a partial order or DAG is called a topological sort of the DAG.

 

In Section 14.5.1, Weiss presents a nice non-recursive algorithm for finding a topological sort of a DAG, and on the way, checking that it really is a DAG, or, instead, has a cycle.  The first step of this algorithm is to determine the in-degree of all vertices in the graph. 

 

The in-degree of a vertex is the number of edges in the graph with this vertex as the to-vertex.

 

Once we have all the in-degree numbers for the vertices, we look for a vertex with in-degree 0.  It has no incoming edges, and so can be the vertex at the start of a topological sort, like CS110.

 

How can we be sure there is a vertex with in-degree 0??  Because if there wasn’t, then we could start a path from anywhere, extend backwards along some in-edge from another vertex and from there to another, etc. Eventually we would have to start repeating vertices.  For example, if we have managed to avoid repeating vertices and have visited all the vertices, then the last vertex still has an in-edge not yet used, and it goes to another vertex, completing a cycle.  Thus the lack of an in-degree-0 vertex is a sure sign of a cycle.  The DAG shouldn’t have any cycles, of course.

 

OK, we have the very first vertex, but what about the rest?  Think recursively!

 

We remove the first vertex from the graph and find the new first vertex for the rest, and so on.

 

But wait, our graph is disappearing.  Let’s be kinder to the graph.  What we can do is remove the in-degree counts for the edge, not the edge itself, and have the same effect.

Calculating in-degrees for all vertices (first step of topological sort algorithm) as a Graph app example.

 

First of all, how can we deliver these results to the caller?  We know each vertex has V-value to identify it.  So for each V value, we have a count of in-edges, an integer.  A Map of V to Integer will do the trick.

 

With JGraphT, all we really have to do is use the type DirectedGraph, and bingo, it has an inDegreeOf(V) method to give these numbers.

 

Pseudocode for Topological Sort by in-degrees, assuming graph is a DAG

 

Calculate in-degrees for vertices and put them in a Map m from vertex to Integer.

Create a queue q and enqueue all vertices of in-degree 0.

Create an empty list t for top-sorted vertices.

Loop while q is not empty

  Dequeue from q a vertex v and append it to list t

  Loop over vertices w adjacent to v

        Decrement w’s indegree (i.e. decrement value in Map m)

        If w’s indegree is 0, enqueue w in q

Return top-sorted list t

 

That’s pretty straightforward, and is one way to do the topological sort problem on pa5.

 

What happen is there is a cycle?  Does this go into an infinite loop, or what?  The presence of a loop means in-degree contributions of one for all members of the cycle, so the cycle protects the whole group from being put on the queue.  For example, consider A -> B -> C -> A and see in-degree = 1 for all vertices.  Each pass of the loop dequeues an element from the queue, so all that can happen is that the queue goes empty with the cycle members still un-enqueued.  Thus a counting trick will be sufficient to add cycle-detection to this algorithm: simply test if the size of t, the delivered list, is less than the number of vertices in the graph.

 

Cycle detection is sometimes all we want.  This algorithm is O(|E|), which is as good as it can be, so this is a good cycle-detection algorithm.  Just drop the list of top-sorted vertices if you don’t want the topsorted result.

 

Weiss has related code in Fig. 14.32, method “acyclic” of his Graph class, using direct access to private data of his Graph class.  Note that our algorithm is a client of our (more powerful) Graph class.  We don’t need access to private data of Graph to loop through vertices or loop through vertices adjacent to a given vertex.  His code doesn’t need the Map to hold in-degrees, because he has put an extra field in his vertex class called “scratch” for extra numbers for various algorithms.  However, this is not clean programming. Our use of Map from V to Integer is fast and appropriate.  It allows us to associate more information with a vertex whenever we need to. 

 

Graph Traversals  (not in Weiss, but should be)

 

Recall tree traversals from CS210.  For a quick review, look at Fig. 18.23.  In the preorder traversal, we start at the root, and plunge down the leftmost side, then back and down the next child of root, down, down, back up, back up, next child, down, down, back up 3x, no more children of root, done.  You should also know the postorder and inorder, but the preorder is the most important and the one that generalizes to “depth first search” of a digraph.

 

DFS—depth first search

 

DFS can be done for an undirected graph as well as a directed graph, but we are most interested in the directed case for pa5.

 

DFS of Figure 14.30, starting from V0, etc. where we use the lower numbered adjacent vertex first. (We need some way of deciding which edge to visit first.  In implementations, we simply follow the order of elements on the adjacency list or edge-set iterator for the vertex.)

 

See how we can follow out-edges down and down:

V0 -> V1 -> V3 -> V4 -> V6 -> V5

We’re finally stuck with no place to go at V5.  We can backtrack back to V6, back to V4, back to V3, see another adjacent vertex, V6, but already visited, back to V1, see another adj vertex, V4, but already visited, back to V0, see another adj vertex, V3, but already visited, done.

We see one unvisited vertex left in the graph, V2, so start again: V2, all adjacent vertices already visited, done.

 

Final DFS visitation order: V0, V1, V3, V4, V6, V5, V2

 

DFS can be used to find a topological sort, but clearly not directly from the DFS visitation order, since V0 -> V5 is OK for forwards top-sort order but V2->V5 is backwards.

 

The trick is to “output vertices on the way back”, that is, after the visitation of the outbound edges. This first outputs V5, then back to V6, output V6, back to V4, output V4, back to V3, output V3, back to V1, output V1, back to V0, output V0. Start another tree: Output V2.

 

Got output of: V5, V6, V4, V3, V1, V0, V2: backwards top-sort order!

Resulting top sort order: V2, V0, V1, V3, V4, V6, V5. Same as before (although this is not required of course, since there can be multiple different topological sort orders for a graph).

 

Pseudocode for DFS.java:

top-level code:

   Set up Set of “unvisited” vertices, initially containing all the vertices.

   for each vertex v

       notify DFS tree is starting (send null vertex with notification)

       if (v is in unvisited)

   dfs(v)

 

 

dfs itself (“traverse” in DFS.java)

   dfs(vertex v)

      notify DFS visitation of v (send v with notification)

      remove v from unvisited set

      for each vertex n adjacent to v

         if (n is in unvisited)

            dfs(n)

     // instead, notify here for top sort, *after* the recursive calls

Top Sort by DFS

 

We can use DFS to do topological sort of an acyclic directed graph, by changing when we do the vertex notification.

   dfs(vertex v)

      notify DFS visitation of v (send v with notification)

      remove v from unvisited set

      for each vertex n adjacent to v

         if (n is in unvisited)

            dfs(n)

         // instead, notify here for top sort, *after* the recursive calls

 

 

For a topological sort of this graph, we need A and D before C, so the DFS order isn’t good with D at the end. It turns out we need to notify the points of last seeing each node, i.e, on the way back from the recursion:

 

Visit A

  Visit B, output B after finding no out-edges

  Visit C, output C after finding no out-edges

   Back to A, output A after finding no more out-edges

---

Visit D, output D after finding no out-edges

 

B, C, A, D

 

We can mark the edges AàB, AàC, DàC on this list and see that they all go right to left, so this is a reverse top sort order.

 

Then the reverse of this is a topological sort.

 

We can do a DFS of any directed (or undirected) graph, and if it’s acyclic, the DFS yields a topological sort. If there is a cycle in the graph, it doesn’t cause an infinite loop because a DFS doesn’t revisit a vertex.

 

Detecting Cycles using DFS

It is possible to enhance DFS to detect cycles.  Color all the nodes white to start, change to gray when visited and black when seen on the way back. Then if you ever try to visit a gray node, it’s part of a cycle, because you are both going inbound to it and are involved in recursion from it, so there’s a path down to you from it.

 

In general a DFS works in phases, finding trees, so the whole thing finds a forest.

 

The first tree is a tree being traversed in preorder.  Once this traversal returns to a vertex, it has classified all vertices it has visited as “downwind” from it, i.e. belonging to the right of it in top sort. (And no edge can be inbound from those, because of the acyclic requirement.) The extreme is V0, the last reencountered after edges in the first tree, and it should be first in the top sort order for that tree.  The second-to-last vertex re-visited should be second, and so on.

 

Then the following trees in the forest have only inter-group edges back to previous groups, so they need to be ordered last-first in top-sort order.

 

The ability of the DFS to turn a graph into a forest of trees is useful in many algorithms.  Trees are a lot easier to work with than general graphs.  Note that a graph does not have to be acyclic to do a DFS.  It’s just that you can only get a top sort out of it if it’s acyclic.

 

Tree Edges

The trees generated by a DFS are using a subset of the edges for the trees. These are known as tree edges.

When we see an outbound edge in DFS, and use it to visit another vertex, that’s a tree edge. If the vertex is already visitied, it’s a non-tree edge.

So the ABCD graph has two tree, two tree edges for first tree, none for second.

BFS Breadth-First Search/Traversal

 

Sometimes we want to visit all the adjacent nodes of a node before going on to look at the further nodes.  This is called a breadth-first search.  We visit direct neighbors of the start node, then neighbors of neighbors, then neighbors of neighbors of neighbors, etc.

 

For Figure 14.30, an acyclic graph :

 

visit V0: we see that V0 has two outbound edges, to V1 and V3

consider edge V0-V1 (tree edge, since V1 unseen. Consider V1 seen now, but not yet processed)

consider edge V0-V3 (tree edge, V3 now seen)

visit V1: we see that V1 has two outbound edges, to V3 and V4

consider edge V1-V3 (non-tree edge, since V3 seen before)

consider edge V1-V4 (tree edge)

visit V3: we see that V3 has 3 outbound edges, to V4, V5, and V6

consider edge V3-V4 (non-tree edge, since V4 seen before)

consider edge V3-V5 (tree edge)

consider edge V3-V6 (tree edge)

  (now finished with neighbors of V0, start looking at neighbors-of-neighbors, all previously seen)

visit V4 (...)

...

 

We end up with a BFS with 3 levels, V0 on the top level, V1 and V3 on the second level, and V4, V5, and V6 on the third level.  V2 was never visited.  As in DFS, we have a tree defined by the BFS using a subset of edges from the graph. 

 

The nodes at the second level are all one hop away from the source node, V0.  The nodes on the third level are all 2 hops away from V0.  Thus we are finding how far, in hops, the various nodes are from V0.  V2 is an infinite distance from V0, since we can’t get there.

 

In Weiss, see BFS mentioned on pg. 542(486), in Sec. 14.2, unweighted shortest-path problem.

 

Programming a BFS

As we are doing a BFS (or DFS) we are defining a tree, and we can think of the “lower” nodes as children of their parent node in the tree.  For the BFS, we find the children of a node, and then go back over them to find their children.  We can put them on a queue as we find them, and when we run out of children, go back and dequeue them and work on their children.  We’ll see this algorithm soon.

DFS vs BFS:

 

  • DFS is like preorder tree traversal, plunging further and further from the source node until we can’t go any further, then back.   Can do cycle detection, also top sort for acyclic graph.
  • BFS is the opposite: explore nodes adjacent to the source node, then nodes adjacent to those (that haven’t been visited yet), and so on.  Good for finding all neighbors, all neighbors of neighbors, etc.—hop counts. (Can be used for cycle detection, easy in undirected case, harder in directed case. We’ll stick to DFS for cycle detection.)

 

Back to Weiss Sec. 14.2: Unweighted Single-Source Shortest-path Algorithm

 

Looking for paths with fewest edges from source node to destination node, in some digraph, not necessarily acyclic.  Fix the source node and consider all possible destination nodes.

 

Fig. 14.16 shows the starting point, 14.17 the 1-edge paths are found, and in 14.18, the 2-edge paths, and finally in 14.19, the 3-edge paths.  Looks pretty easy.

 

Note this is using a BFS.

 

But we don’t want to search the whole graph looking for the next nodes to work on.  It’s clear that the nodes marked for the first time are the ones that soon should be reexamined for the next hop.  So to be efficient, we can save them on a queue, and then pull them off the queue for the next hop.

 

At one point, the queue will have all the neighbors of the start node. Sometime later, it will have all the neighbors of neighbors.  But in between these times, it will have some neighbors and some neighbors of neighbors, all behind the neighbors in order, so they won’t be handled until all the direct neighbors are processed.

 

Needed fix to Weiss, new edition, Fig. 14.21

That’s what is shown in Fig. 14.21.  The “eyeball” nodes are the lightly-shaded ones in 3rd edition, unmarked in 4th edition, that is, V2 in panel 2, V0 in 3, V5 in 4, V1 in 5, V3 in 6, V4 in 7, and V6 in 8.  We see these are just the nodes in BFS visitation order.

 

Now we consider how to turn this into an effective algorithm.  After we’ve marked all the 1s for one-hop nodes, we need to revisit the one-hop nodes to find the two-hop nodes.  That is, we need to save them and later access them.  We can use a simple queue of nodes for this.

 

 

The pseudocode for Unweighted Single-Source Shortest-path Algorithm, is:

 

Given a digraph G and source node S.

Set up storage for a distance for each node, Dn:

   For each node n, make Dn be INFINITY.

Set DS = 0

Create queue q.

Enqueue source node S on q.

while the q is not empty

    Dequeue a node from q and call it v (the eyeball node)

    Loop though nodes w adjacent to v

          if Dw is INFINITY (not yet marked)

                set Dw to Dv plus 1

                enqueue w on q
