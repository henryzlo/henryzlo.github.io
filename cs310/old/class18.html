Running a game: PlayOneGame

 

Games and Players come together in PlayOneGame objects.  Each PlayOneGame object HASA Game and 2 Players, one of which ISA HumanPlayer and the other ISA a ComputerPlayer in the normal way we use it (but it can be 2 ComputerPlayers or 2 HumanPlayers). We see that the Player that goes with Game.FIRST_PLAYER is Player “one” of PlayOneGame, the player that takes the first turn, and the other is “two”.   The Player “one” can be human or computer, depending on our answer to the question about going first.

 

Thus the correspondence between PlayerNumbers and Players is held in the PlayOneGame object. It was decided on in PlayGame.

 

Naturally, we need code to run a game.  All this needs is a loop of calls to findbest of Player and make of Game.  This is in PlayOneGame.java, in the games package.

 

We made a picture of the objects: PlayOneGame has refs to a Game/Easy (say), meaning a Game that is actually an Easy Game, and refs named one and two to Players, specifically, say, Player/HumanPlayer and Player/ComputerPlayer/Backtrack. The HumanPlayer subclass has a ref to an object that knows how to do UI, but PlayOneGame only sees types Game and Player, so it has no knowledge of the UI object or the specific game being played.  This shows the power of abstraction: PlayOneGame can play the game without knowing much about it: what it needs is provided by the Game API, and similarly with the Players.

Playing the game. PlayOneGame has a loop which calls findbest of Player to get the next move and then makes the move via the Game API.

 

Well, it’s not quite that simple. In PlayOneGame.java, we see calls to makeObservable rather than to make, so the basic play-game loop has calls to findbest and makeObservable.  What is this all about?

 

In Game.java, we can read makeObservable and see that it calls make and then notifyObservers().  Who are these observers we’re notifying?  They are the timing module and the UI.  Both of these modules are interested in the event of a move happening. 

·   The timing package wants to separately time each player, so wants to know about moves.

·   The UI wants to update or redraw the board after each move, so is also interested in moves. 

 

 

The Observer Pattern

 

See pg. 51 of the handed-out excerpt from F&F for the general picture.  One object holds state and notifies others about changes in that state. In our case, the Game object holds the game state, and moves change that state, so each move triggers notifications to the “dependent objects” or Observers. In our case, there are two observers, the timing package and the UI object.

 

The F&F coverage goes on to show how to implement the Observer pattern from scratch in Java. Then they start covering the JDK support. The excerpt has pages 64-65 on that coverage.

 

The Observer interface and the Observable class are in the standard JDK. Together they give us the “observer pattern”.  See pg. 64 of F&F for a UML class diagram.  For us, replace WeatherData with Game, GeneralDisplay with GameTimer, and StatisticsDisplay with UI (GameView).

 

In this pattern, the Observable object sends out “notifications” when it changes to registered observer objects, by calling notifyObservers(). The observer objects then can check further by calling various methods of the observable. The Observable doesn’t know anything about the observers, so you can imagine that the observers are (roughly) spying on the observable, interested in when it changes.

 

The observer pattern is important to proper software engineering of GUI apps, so you will find it discussed in that context in many places.  Here we are using it even for a line-oriented app.

 

Here the Game ISA Observable, as you can see from the start of Game.java. The game state changes occur because of moves, so the method makeObservable  calls setChanged() and notifyObservers(). Objects interested in tracking the moves of a Game can register themselves as Observers by calling game.addObserver().

 

The Game itself does not need to know about these specific needs once it has become an Observable—any interested object can sign up for notifications. This keeps the Game code simpler than if it had to know explicitly about the UI object and the GameTimer object, i.e., have refs to these objects in its own fields. Instead, class Game just extends Observable, and the JDK code of Observable manages a registry of observers for it.

 

When the timing package is notified about a move, it stops timing one player and switches over to timing the other player.  When the UI is notified, it repaints the representation of the game for the user, because the move has changed things.  Let’s not worry about implementation details of this mechanism now, just appreciate the results.

 

For the interested, here are some more details:

1.      Setup: Game extends Observable, itself a concrete class, bringing to Game the implementation of the notification-registration system and the notification-sending system:  The calls used in game classes are addObserver(Observer o), setChanged(), notifyObservers(),  notifyObservers(Object arg);

2.      Setup: the Observers are the timing package and the UI. They each need a class that implements Observer in order to receive notifications (via calls to method “update” in the Observer). The UI base class GameView itself implements Observer, and in its own initialization, calls g.addObserver(this), adding itself to the registered Observers of Game. The top-level timing class GameTimer also implements Observer, and also adds itself by g.addObserver(this). Each of these Observer classes have a method called update that is ready to handle a notification.

3.      Operation: when a move is made via makeObservable in Game, Game code there calls setChanged() and notifyObservers(m), passing the move as the arg.

4.      Operation: Each Observer gets a call to its update method, with the move object as an argument. It does the action desired: In GameTimer, it checks about switching which player to charge time to.  In the UI, it shows the user the new board.

5.      Operation, case 0: when the game is started, Game.initObservable calls notifyObservers(), no arg.

6.      Operation, case 0: Each Observer gets a call to its update method, with move=null, and does any appropriate action for the start of the game.

 

 

Summary on PlayOneGame:  PlayOneGame loops through the moves of a game and, by making moves in the Game, indirectly notifies appropriate parties about the moves.  Again note that it knows the Game only as Game, players only as Players, yet still can play the game.

[optional coverage on GUIs] We’re not officially worrying about a graphical user interface (GUI), but here’s a note about it for your general information.  This idea of notifying the view module about changes to the “model” is part of the MVC architecture, for model-view-controller, an architecture originally developed for pre-web (everything on one computer) GUI treatments. The model is the “business logic” or whatever we are actually trying to accomplish, here the game-playing software.  The view is the display to the user, and multiple views of the same model are supported in general. The controller interprets user input and issues appropriate calls to the model or the view.  The model (an Observable or equivalent) notifies the view about updates to the data of the model, via the observer notifications. 

 

In web terminology, following the above architecture would mean that the back-end (the model) notifies changes to the view (the browser.)  But browsers are basically pull-based (user requests each action), so this doesn’t work so well.  The MVC architecture has been flattened for browser-based web UIs to a linear arrangement of model connected to controller connected to browser (by HTML/HTTP.)  This is also called “Model 2.” See http://www.billsgames.com/tic-tac-web/ for an example of a browser-based (no Java applet) Tic-tac-toe game.  Note how the whole board is repainted to show one move, and how you have to push a button to get a new picture.  That’s because the browser (using only HTML/HTTP) has to request something to get a new web page.  If you download a Java applet, of course it would be able to push new data to the screen whenever it wants to. Similarly with other programming methodologies that run Javascript or specialized programs, not just the browser, on the client computer.

 

Tictactoe has a Swing full-MVC GUI in gamesui. The controller is in GameController, and it looks very much like PlayOneGame, and replaces PlayOneGame for the GUI case. The difference is that a GUI code can’t call for user input. Instead, it must react to the user-input event when it happens. To bridge this gap, findbest is given another way to return: it can return null, and that means that the GUI will submit the move later on in time. The GUI controller accepts the new move in a separate call, and does the makeObservable call into the Game.

 [end of optional coverage on GUIs]

 

 

Backtracking Algorithm

 

Pseudocode for minimax search for best move (a backtracking algorithm.)  Weiss uses low values for positions good for humans, high for computer.  It’s easier for us to use a high value (1) for game states good for player 1, low value (-1) for player 2, since the Game doesn’t know human from computer players (it has no need to.)  A draw is 0 in this system.

 

The minimax search we need for getValue(Game g) is as follows, working from the code in Weiss, pg. 337(295).

 

If the base game state is a done game,

return the appropriate value immediately

 

Determine the current player# from the base game state

Initialize a running min/max, to worst possible outcome for player#

 

Loop through moves from this base game position.

For a certain possible move:

   Make the move on a copy of the game state

   Get the value for this new game position by calling getValue

   Use the value to update the running max or min, depending on player#

Return the min/max value

 

 

                * you can also return the move that goes with the best value, as Weiss does, but see more discussion below

 

Example of execution: getValue(Game g) where g is a state where Player1 is next.

    player# is 1

    Since 1 wants high values, we need a running max, initialized at -1, the worst possible value for player 1

   loop through moves from this g position

      make the move on a copy of the game state

      call getValue to get its value

      Update the running max

   return running max

 

Suppose that getValue returns 0, -1, 1, and 0 from the various recursive calls. Then the running max will end up at 1, and 1 will be returned from this call. It means that even with Player2 playing perfectly, there is a win for Player 1 from g, by taking the #2 move. However, in this simple getValue, we don’t report the move, just the value.

 

But at top level we do need to find a move. We can use getValue as a helper method to the top-level findbest. In findbest, we also loop through the moves, in this case calling getValue, not findbest, so it’s not itself recursive. The move that has the best value is the one to return from findbest.

 

What about the side parameter in Weiss’s method?  It's not really needed with our Game ADT because the game state knows whose turn it is and that can determine whether to do the max or min.  We do need a Game parameter, however, because this code is outside the Game class. 

 

You can see there are two ways to implement findbest.  If the helper returns only the value, not the move for it, the top-level findbest code can do its own min/max to figure out the best one.  The advantage with this approach is a simpler getValue, since it simply finds a value for a certain Game state.  The disadvantage of this approach is the distribution of the min/max code—it’s nicer to have it all in one method.  The simpler getValue is easier to upgrade to dynamic programming.

 

Dynamic Programming for findbest, also in pa4

 

Now we add a map of values for various game positions, so we avoid recomputations. This is a map of Game to Integer. Since Game is required to have equals and hashCode, we know we can use a HashMap.

 

Be careful about object sharing of mutable objects used as keys when working with the Map of Game to Integer here.  We need the Game objects to keep the same value once inside the Map.  That’s required (in the fine print) to keep the Map lookups working properly, with HashMap or TreeMap.  So make copies of Game objects as needed.
