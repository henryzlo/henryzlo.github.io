JDK Sorting

 

Sort: The JDK has many versions using primitive types, like static void sort(int[] a).

 

Here are the JDK full-array sorts for object arrays

static void

sort(Object[] a)

static

<T> void

sort(T[] a, Comparator<? super T> c)

 

 

Sort: Weiss, Sec. 6.4.4, has these methods, plus a comment that they have been made generic:

void sort(Object [] arr);  //static method, in JDK

void sort(Object [] arr, Comparator cmp); /// raw type Comparator, pre-Java-5 (but works of course) Not in JDK as such.

 

The version without the Comparator uses “natural order” of the array elements, i.e., calls compareTo of the element type to compare elements. Thus the elements need to be Comparable, that is, the element type implements Comparable<ElementType> in the generics setup.  Or the old Comparable works here too.  Either way, the elements have a compareTo method that gets used by the sort.

 

What is that <? super T> clause?

The Comparator<? super T> specifies that T ISA Comparator<Y>, where Y is T or any superclass of it.  This allows the use of a Comparator<Y> implemented at the top of an inheritance hierarchy (i.e., for the base class Y) to compare elements of subclass type in the array.  For example, we commonly use a unique id for equals, hashCode and compareTo across a hierarchy, and only want to implement it once in the base class. See Sec 4.7.4  for more info on <? super T>.

 

Ex: Employee has subclass Manager, we have Comparator<Employee> cmp, we can sort an array of Managers with it.

 

Why doesn’t the JDK have both sort(Object[] a) AND sort(T[] a)? 

Answer: Because it can’t. Java does not allow us to have two methods in one class that overlap in applicability. You can see that the Object version is a special case of the generic version, so they cannot both exist in the API for the class.  In fact the Object version is “the erasure” of the generic version.  Faced with this choice, the JDK designers decided to go with the simpler Object version, probably to avoid scaring old-timers too much. Anyway, the Object version works great for any object type. 

 

Summary: If you want to use natural order, the Object[] version is fine, and the JDK can’t also offer the generic version.

 

Is there a downside to this?

 

Of course the compiler doesn’t have enough information to prevent you from trying to sort elements that don’t have a compareTo method. If you do this, you’ll see a runtime exception when sort tries cast the element to Comparable in preparation to call compareTo.  (You could check for this before sorting by “e instanceof Comparable” test.)

 

If you want to specify a Comparator, the generics version comes into play, and it also handles raw Comparators if necessary.

 

Ex. sort a in natural order (where elements of type E implement Comparable<E>)

     Arrays.sort(a);

  --or—

  import static java.util.Arrays.sort;

  ...

  sort(a);

 

Ex: sort a in reverse order:

 

    Arrays.sort(a,Collections.reverseOrder());

 

Note: in Weiss Collections, pg. 244(216), cross out the static class DefaultComparator—that is not in the JDK. Of course you can implement it yourself.

 

Subarrays: sequences of array elements, a[i] to a[j], j >= i. These can easily be sorted too--

  JDK ex: static void sort(int[] a, int fromIndex, int toIndex)

The range to be sorted extends from index fromIndex, inclusive, to index toIndex, exclusive. (If fromIndex==toIndex, the range to be sorted is empty.)

 

JDK Searching (binary search of a sorted array)

 

Fig. 6.15, pg. 247(219): the first version of binarySearch is not in the JDK with AnyType, only the second version, plus many other versions with primitive types and Object

 

Here are the JDK full-array binary searches for object types (there are also subarray sort versions and primitive-array versions):

 

static int

binarySearch(Object[] a, Object key)

static <T> int

binarySearch(T[] a, T key, Comparator<? super T> c)

 

The version without the Comparator uses “natural order” of the array elements, i.e., calls compareTo of the element type to compare elements. Thus the elements need to be Comparable, that is, the element type implements Comparable<ElementType> in the generics setup.  Or the old Comparable works here too.  Either way, the elements have a compareTo method that gets used by binarySearch.

 

The version with the Comparator uses its compare(x,y) to compare elements x and y.  Because of type erasure, it also works for old-style Comparators.

 

 

We can call the simpler form easily:

 

 String[] arr = {“a”,”b”};  // sorted array

 int index = Arrays.binarySearch(arr,”b”);  // will return 1

 int index1 = Arrays.binarySearch(arr,”c”);  // will return -1, not found

Binary Search: look at recursive version, pg. 308(266), and non-recursive version, pg 209 (185), which we expect to be slightly faster in practice.  But they work the same way.  A simple case of divide and conquer.

 

Both of these are generic methods that can take any object array AnyType [], and an AnyType x to find, and return the int index of x in the array, or NOT_FOUND (-1). This exact form is not in the JDK, as discussed above, but the form with the erased types is there, usable for any array.

 

Mergesort: variations of this can do in-memory and external sorts.  It also gives us a good example of divide and conquer.  And of course it’s a good sort, O(NlogN).

 

Idea of merging 2 arrays—look at pp. 313-316, code pg. 316-317. 

 

Performance:  (we did this earlier, so not covered in class)

              T(N) = 2*T(N/2) + O(N)            

                   = 2*(2*T(N/4) + O(N/2)) + O(N)

                   = 2*(2*(2*T(N/8) + O(N/4)) + O(N/2)) + O(N)

                   = 2*2*2 T(N/(2^3)) + O(N)+O(N)+O(N)

                   = ... = 2*2*…*2 T(1) + O(N) + O(N) + … + O(N)

                    2 multiplied by itself log N times = N

                     log N terms of O(N) = O(NlogN)

      

You can see that the merge being O(N) is crucial to this calculation, and drives the result.  This is divide and conquer in action, giving us a decent sort algorithm.  But it’s not the best in practice for memory sorts.  It is more important in practice for disk sorts.  The merge action is easily done between two sorted files—it is very sequential in nature.

Greedy Algorithms and Dynamic Programming

 

Intro’d in Sec. 7.6 of Weiss.  As we all know, a greedy person grabs everything they can as soon as possible.  Similarly a greedy algorithm makes local decisions that appear to be the best thing to do at each step.

 

Change-making greedy algorithm for “change” amount, given many coins of each size:

Loop until change == 0:

  Find largest-valued coin less than change, use it.

  change = change – coin-value;

 

Example of change-making with the fewest number of US coins—have 1, 5, 10, 25 unit coins to work with.  Clearly we want to mainly use large-value coins to minimize the total number.  So for 27 cents, clearly we can’t do better than 25 + 2(1). 

 

What about 89?  The greedy algorithm says use as many 25s as fit, 89 = 3(25) + 14, then as many 10s as fit in the remainder:  89 = 3(25) + 1(10) + 4, no 5’s fit, so we have 89 = 3(25) + 1(10) + 4(1), 8 coins.

 

It’s annoying to have the 4 pennies, but we can easily see that it’s required, since all the other coins are multiples of 5 cents.  The number of pennies is always going to be n%5 here.

 

In fact this greedy algorithm gives the optimal solution for this coinage.  Nice situation.  We don’t have to remember special cases in daily life.

 

But with different coinage, the greedy algorithm doesn’t find the optimal solution.  The book gives the example of US coinage with an additional 21 cent piece.  Then 63 = 3(21), but greedy says use 2 25s, 1 10, and 3 1’s, a total of 6 coins.  The 21 cent piece is hidden a little under the 25 cent piece. 

 

Looking ahead:  Doing the wider search to handle problematic coinage, where the greedy algorithm doesn’t work.

 

The first algorithm, on pg. 331 (289), tries all the different ways to subdivide the amount into two parts and the use recursion on those parts. This is double recursion, and that’s a red flag.  Since it doesn’t make enough progress in cutting down the problem size in the recursive call, it turns out to be “bad” recursion, with resulting exponential running time.

 

But we will be able to fix this terrible performance by using dynamic programming. In dynamic programming, we save results as we go and reuse them.  For example, when we first discover that 68 cents takes 5 coins in a certain coinage, we save that fact (68 -> 5) in a Map, and then use it when we’re looking at how to make change for 74 cents, say.
