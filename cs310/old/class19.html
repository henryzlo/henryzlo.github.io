Chap 14 Directed Graphs (and mention of undirected graphs)

Look at Fig. 14.1, pg. 528(448).  This is a directed graph, or digraph.  An undirected graph has no arrows for edges, just lines denoting two-way connections.  It’s OK to have two arrows between V0 and V1, one in each direction.

 

Terminology

"Representations" for graphs—this means implementations of graphs.   Let's see how they work:

--adjacency matrix: take too much memory for sparse graphs, the most common kind in apps

--adjacency lists: the better way in most cases, since they don’t waste memory in the case of sparse graphs. Can use map instead of list for faster access.

 

Small example: directed graph with vertices A, B, C (also known as 0, 1, and 2), edges AàB, AàC, CàB

This graph has a 3x3 adjacency matrix adj[3][3] with all zeroes as entries except for 3 ones in adj[0][1] for AàB,  adj[0,2] for AàC, and adj[2,1] for CàB.  

Alternatively, this graph could be represented with adjacency lists, one for each vertex:

Vertex A: [B]à[C]     (vertices B and C are adjacent to A)

Vertex B: null

Vertex C: [B]

But note that the “adjacency list” could be another container of adjacent vertices, such as a set, since the elements of the list don’t maintain important order information.  We could still call it an adjacency list.

 

Look at Fig. 14.2.  Here the nodes are indicated by the numbers at the left.  So 0 denotes V0 of Fig. 14.1, and it has a 2-element adjacency list, with one element for its adjacent node V1 and another for its other adjacent node V3, with edge costs of 2 and 1 in parentheses.  Similarly, nodes V1 and V2 each have two out-arrows, and thus two adjacent nodes and two elements on their adjacency lists.  Node V3 has 4 out-arrows and four elements, and so on.

 

We need a Graph Package.  The JDK has nothing, and the book doesn't give a decent one, so we need to work on it, or find another: JGraphT, an open source package, but let’s work up to that.

 

Go back to definition, pg. 528(472): "A graph consists of a set of vertices V and a set of edges E.  An edge is an ordered pair of vertices (v,w), or a triple if there’s a weight on the edge, (v, w, W)

 

The graph model:  OK, we have vertices and edges to work with, but they aren't quite equally prominent.  We see that an edge is a connection between two vertices, and derives its identity that way—it's not the edge "Fred" but the edge from v to w.  The vertices are expected to be individually identifiable.  Often we just identify them with integers, as in Fig. 14.1.  That graph has 7 vertices, id'd V0, V1, V2, ...V6 and 12 edges:, listed in the middle of pg. 528(472).

 

We already have Sets, so we could just follow the def and represent G with a Set of vertex-ids and a Set of vertex pairs, a pair just being an object with 2 vertex-id members, like Spot in the Grid problem.  Is this what we want?

 

As usual, the question boils down to what we want for operations on graphs.  We want to be able to follow paths in the graph and find all the ways out from a given vertex.  The set of edges is made up of subsets of edges leaving each node, and this subset structure is important.

 

A vertex needs to be unique, but its actual type is up for grabs. With generics, we can parametrize the vertex type, so handle graphs with String vertices, or Integer vertices, or ..., with the understanding that the app needs to come up with unique Strings, or unique Integers, for actual vertices.

 

Note that Weiss does not take this approach, and has plain Vertex and Edge classes, shown in Figures 14.6 and 14.7. However, we are not going to use his Graph code. Instead, we’ll use the JGraphT package.

 

An edge does not need an app-supplied unique id, because its main identity comes from its connection of vertices. The Graph object can be responsible for knowing the connections.

 

For a graph of Airports: suppose our app has Airport objects...

 

public class Airport {

       private String name;  // “BOS”, “LAX”, etc.

       private int timezone;

          ...

       public boolean equals(Object x) {...}

         

         ...

}

 

We want a graph of airports, showing the possible flights between them.

 

The simplest approach is to make a graph of String vertices, and use the airport names. We can always map from the names to the other info.  Alternatively, we could use vertices of class Airport, with equals based on name.

 

For a flight from “BOS” to “LAX”, we want an edge. That edge may not need an app type. But sometimes it is useful to have an app-defined edge type. What is a Graph package to do?

 

JGraphT solution: Graph<V, E>, but E can be “DefaultEdge” if you don’t want to bother with your own edge type.  You do need to come up with type V, often String or Integer.

 

JGraphT Graph<V,E> Choices for the app in modeling edges, requirements on types V and E

So if an app chooses Graph<String, String>, it is promising to come up with unique Strings for vertices, and also unique Strings for edges. The Graph object will remember all these ids and support access by them.  If an app chooses Graph<String, DefaultEdge>, it is only promising to come up with unique Strings for vertices. The app can find edges of interest by how they are connected to its vertices. The Graph itself will come up with unique DefaultEdge objects for edges, which the app can use or ignore.

 

Tech note: in fact, String edges are hard to work with in a way that is pretty subtle. If you create a graph as below with String replacing DefaultEdge, that is, with “String.class” passed as an argument to a graph concrete class constructor, you will get a graph usable in many ways, but g.addEdge(v,v) won’t work properly, because it requires an internal edge creation that uses the default EdgeFactory, that in turn uses String.class.newInstance(), which always returns the nullstring, not unique ids as needed. DefaultEdge does not have this problem because its equals() is Object equals(), and all the new instances are unique under this equality. If you really want String edges, you should write an EdgeFactory to produce unique strings, and pass an EdgeFactory object to the concrete graph class constructor of that form.

 

Here is the basic setup for a tiny directed graph of airports, using String names “LAX”, “BOS”, etc. for vertices and DefaultEdges:

 

Graph<String,DefaultEdge> airGraph = new                             SimpleDirectedGraph<String,DefaultEdge>(DefaultEdge.class);

airGraph.addVertex(“LAX”);

airGraph.addVertex(“SFO”);

airGraph.addVertex(“BOS”);

airGraph.addEdge(“BOS”,“LAX”);

airGraph.addEdge(“SFO”,“LAX”);

 

Alternatively, we can use Airport for the vertex type, assuming it has equals and hashCode that are based on the name, so that name is still the unique id.  These requirements are the same as what we need for HashSet<Airport>, assuming that we are using name for identity.

 

Graph<Airport,DefaultEdge> airGraph = new
              SimpleDirectedGraph<Airport,DefaultEdge>( DefaultEdge.class);

Airport a1 = new Airport(“LAX”, 8);

Airport a2 = new Airport(“SFO”, 8);

Airport a3 = new Airport(“BOS”, 5);

airGraph.addVertex(a1);

airGraph.addVertex(a2);

airGraph.addVertex(a3);

airGraph.addEdge(a3,a1);

airGraph.addEdge(a2,a1);

 

JGraphT: an open-source package.

Note that addEdge returns an edge object, but we didn’t bother to assign it to a variable in the above code. There’s no real need to, because the Graph knows all about the edges. For example, we can do “airGraph.containsEdge(a2, a1)” to see if there’s an edge from a2 to a1.
