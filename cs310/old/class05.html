Database-like lookups

Often in applications we use database-like data, that is, a bunch of objects with the same fields, and one of them is an identifier (or key.)

 

Let's next look at a little example of that sort.  Suppose we have a simple inventory system where part names are keys to inventory objects holding  (name, QOH, bin#), for ex.,

 

(“pencil, 120, 42)          120 pencils in bin 42

(“tape”, 44, 11)              44 rolls of tape in bin 11

…

Suppose the file inventory.dat looks like this:

 

120|42|pencil

44|11|tape

10|47|quadrille-pad

 

Here we are using “|” as a data-field delimiter.   Another common delimiter is “:” and also “,”.  Space is not so useful because it occurs to commonly inside data fields.  Note that Scanner and String.split() can handle any delimiter or alternative delimiters, specified by a regular expression—more on this in a moment.

 

Once this data is pulled into memory, our users want to look up facts, for example, how many pencils there are.

Query: how many pencils?

 

First, let’s set up an object for each item:

=============================================================================

public class InventoryItem {

   // constructors...

public InventoryItem();

public InventoryItem(String name, int bin, int qoh) { ... }

 

  // getters and setters... (best to have no setter for the name, since it’s the key)

  public String getName() {...};

  public int getBin() { ... };

  public int getQoh() { ... };

  public void setQoh(int newQoh) {...};

 

  private String name;

  private int bin;

  private int qoh;

}

===============================================================================

Note: This class can be called a “Java Bean”, with properties name, bin, and qoh. Each property has a getter getX and/or a setter setX, where the capitalization is converted as you see. The fields inside are all private. There is a no-args constructor. This makes it a Java Bean in most definitions, although sometimes additional requirements are expected.

 

The powerful IDEs like eclipse and netbeans have assistance with creating a Java Bean. Type in the fields, select them, and right click and find something like “create getters and setters”.

 

A Java Bean is usually just a data-carrier, but in some cases has additional functionality provided by methods other than just getters and setters.

 

Now suppose we want to put all this data in memory—should we use a Set or a Map?

 

Let’s try Set of InventoryItem, and see what happens—

         0. Create Set:  Set<InventoryItem> invDB = new HashSet<InventoryItem>();   Need to add an equals, hashCode method to InventoryItem for this, or leave both out.  OK, do that.

1. load the Set: no problem, loop over

    InventoryItem newItem = new InventoryItem(name, bin, qoh);

    invDB.add(newItem);

2. do query: how many pencils?

    We can find out if there is a record for “pencil” by invDB.contains(“pencil”), but there’s no way to pull out the whole  record for pencil based on key-match.  That’s the problem with using Set for this app.  We need to use Map.

(We could iterate over the whole Set, and find the match, but that’s O(N), and we want something faster.)

 

Second try Map of String name to InventoryItem, and see what happens—

0.Create Map:  Map<String, InventoryItem> invDB = new HashMap<String,InventoryItem>();  

No need for equals/hashCode for InventoryItem, only for String, already there.

1.load the Map: loop over

    InventoryItem newItem = new InventoryItem(name, bin, qoh);

    invDB.put(name, newItem);

2.do query: how many pencils?  No problem, O(log N) for TreeMap, O(1) for HashMap:

    InventoryItem item = invDB.get(name)  // (no cast needed, thanks to generic collections)

 

One concern might be that we have two copies of the name string for each item, because we used one for the key, and another in the InventoryItem.  But in fact Java handles Strings very efficiently and there is no duplication, except for the references.

 

Reading structured data, like this inventory data

 

Looking at the qual solution, we see a pattern:

 

1. get a Scanner on the input file

2. use it to read lines

3. for each line, use split, or another Scanner based on the String line

 

In the inventory example, one line of a file corresponds to one instance of InventoryItem. 

We said the file inventory.dat looks like this:

 

120|42|pencil

44|11|tape

10|47|quadrille-pad

 

So the first call to nextLine() returns String line = “120|42|pencil”. No end-of-line is included, probably because that varies by OS and so causes portability problems if included.

 

String parts[] = line.split(“|”);  // But | is a special character to regular expressions, needs quoting with \

String parts[] = line.split(“\|”);  // But \ is a special char in Strings, needs quoting with \

String parts[] = line.split(“\\|”);  // This works: uses “|” as separator

 

Note: if the file used a comma as delimiter, it would be just line.split(“,”);

But sometimes we receive files from somewhere, so we have to live with some particular delimiter.

 

now parts[0] = “120”, ..., should check that parts.length == 3

 

Or use another Scanner for the line string, with non-default delimiter: let Scanner do the number conversion:

Scanner lineScanner = new Scanner(line);

lineScanner.useDelimiter(“\\|”); // same regular expression

int quantity = lineScanner.nextInt();

int bin = lineScanner.nextInt();

String name = lineScanner.next();

Maps of Strings to Lists

We have looked at Maps of Strings to objects such as InventoryItem, but we can be more adventurous!

 

Look at pg. 498(444) in Xref.java:

Map<String, List<Integer>> theIdentifiers = new TreeMap<String, List<Integer>>();

 

This is a map of Strings to Lists of Integers. Each string in the domain maps to a List<Integer> in the range.

We don’t need to worry about equals/compareTo because String has these beautifully implemented for us.

 

Note: we don’t need a concrete class for List here, for example, new TreeMap<String, ArrayList<Integer>>();. We only need the concrete class when we create this object. Inside <>, we only need the type, i.e., the API we are depending on. Later, when we come to create a List to put in this container, we’ll need to use “new ArrayList<Integer>” or “new LinkedList<Integer>”.

 

If we did need a method of ArrayList or LinkedList not in List, then it would be necessary to use ArrayList or LinkedList type. Always use the most abstract type that provides the API you need. It gives you future flexibility.

 

Let’s see what’s in this, for our example, simplified a bit.

 

1 import java.util.Map;

2 // Xref class interface: generate cross-reference

3 /**

4 * Class to perform cross reference

5 * generation for Java programs.

6 */

7 public class Xref

8

9 {

10    public Xref( Reader inStream )

 

 Whole data structure: Map of String to List of Integer:

“import” --> [1]

“java”--> [1]

“Xref”--> [1, 7]                <-- this means the id “Xref” shows up on lines 1 and 7

...

“public”-->[7,10]

 

Access examples:

Add “java” for the first time.

List<Integer> value = new ArrayList<Integer>();

  value.add(1);

theIdentifiers.add(value);

 

Add “public” the second time:

List<Integer> l = theIdentifiers.get(“public”);

l.add(10)

// done, it’s already in the Map!

 

 

Also: Lookup an ID, get a List, with get().

 

When we get a ref on the List with get, we are obtaining the “live” object inside the Map. Not a copy.  So we don’t have to “put” the List back in the Map after changing it.

 

What about equals/hashCode/compareTo here?  They are only needed for the domain type, here String, so the JDK has done all the work for us. That is very commonly the case: we map from some sort of simple ID in the domain to a more complicated value in the range.
