We can use multiple iterators for one list:

Iterator<Order> itr1 = list.iterator();

Iterator<Order> itr2 = list.iterator();

 

This works fine as long as they both are only reading the list contents.

For example: given a list of Points, find the longest distance between them.

Solution: use multiple iterators

Given List<Point> list

Assume Point has public static method double distance(Point p, Point q)

 

Iterator<Point> itr1 = list.iterator();

Double max = 0;

while (itr1.hasNext())

   Point p = itr1.next();

   Iterator<Point> itr2 = list.iterator();

   while (itr2.hasNext()) {

      double dist = Point.distance(itr1.next(), p);

       if (dist > max)

         max = dist;

  }

         

Note: this calculate distance(x,y) and distance(y,x): consider how to avoid that.

Here is the Order class. Note that we need to override equals to have proper duplicates in the list, that then can be found using .equals. If we remove the override of equals in Order, there will be no duplicates in the list if we separately create each element Order, because each element will have a different reference value.

 

public class Order {

private String foo;

       public Order(String bar) {

              foo = bar;

       }

       public String toString() { return foo; }

       @Override

       public boolean equals(Object x) {

              if (x==null || getClass() != x.getClass())

                     return false;

              Order o = (Order)x;

              return foo.equals(o.foo);  // using String equals

       }

}

 

 

Note there are no concurrent threads here, just concurrent iterators on one List, in single-threaded code. Once you realize what’s going on, it’s easy to fix here: just drop itr, leaving position, get o1 by get(position), though this does lower potential performance with LinkedList, and remove is a similar problem in ArrayList

Performance of LinkedList vs. ArrayList (N = size of list)

 

An ArrayList is a wrapped-up array, so get and set are very fast, O(1).  An append-type add should be fast most of the time, since the array grows by leaps.
But when it causes an array expansion, it’s quite expensive, O(N).  Deletions by remove of an iterator would be slow, O(N), unless it’s known to be at the end.

(ArrayDeque has faster remove)

 

A LinkedList is a doubly-linked list that knows its own size, so get(1) is done by two next’s down the list from BOL, and get(n-2) is done by two previous’s from EOL. 
The most expensive get is get(n/2), the half-way point.  This two-ended fast access is a nice property, and allows us to build stacks and queues on top of LinkedLists with good performance.

 

Linked lists are great for handling inserts and deletes, once the right spot in the list is located.  Remove in an interator is O(1), but the larger task may involve O(N) next’s to get the iterator positioned.

 

Intro to Sets: Sec 6.7

Idea of a set is that each set contains a number of elements, with no duplicates.  E.g., set of integers:  A = {1, 5, 3, 96}, or B = {17, 5, 1, 96}.  Set of strings, {"Mary", "contrary", "quite"}.

 

Sets have no duplicates and order doesn't count.  (vs. List allows duplicates and order does count.)

 

Thus can't have {"Mary", "contrary", "quite", "Mary"}. 

 

In Java, the Set interface is just the Collection interface.  The API isn’t sensitive to the lack of duplicates, only the implementation. 
The implementations (concrete classes that implement Set) in the JDK are the TreeSet and HashSet.  They check for duplicates by using the equals method of the elements (pg. 265(232).)
Additionally, HashSet uses hashCode() and TreeSet uses compareTo (in the simple-constructor case, anyway).

 

Sets of JDK element type (String, Integer, ...) are very easy to use, because the JDK classes all have appropriate equals, hashCode, and compareTo (they implement Comparable<E>).

 

Ex: Simple set app using element type String, concrete class HashSet—pg. 265(231). If we add “hal” again, no difference in resulting Set.

 

Ex: Simple set app using element type String, concrete class TreeSet—pg. 264(230), but to keep it simple, drop the argument to the TreeSet constructor, to make it simply “new TreeSet<String>()”.
Then the Strings are kept in lexicographic order by ASCII code values: “A” < “Z”, “Z” < “a”, “a” < “b”, etc.   Here “bob”, “hal”, “joe” would be printed out.
If we add “hal” again, no difference in resulting Set.

 

Of course a pure set is supposed to be without order, and here we are seeing order imposed by the TreeSet. It’s an extra feature, so the TreeSet gives us a SortedSet. 
We can just ignore the order if we want. The TreeSet gives a high-performance implementation, competitive with HashSet.

 

Set of user-defined objects.  If we use our own class for the element type, we have to make sure equals, and hashCode or compareTo are in good enough shape to work properly when called by HashSet or TreeSet on the element objects.

 

Consistency requirements: equals and hashCode must be consistent, so that if a.equals(b), then a.hashCode() == b.hashCode (see pg. 267 (232)).
Also equals and compareTo must be consistent, so that if a.equals(b), then a.compareTo(b) == 0.

 

Note: equals() always has an Object type parameter, use of @Override. If you forget and put equals(SomeType rhs), you end up with another equals that doesn’t override Object.equals,
causing mysterious problems. Mark this note on pg. 267 (232) along with Weiss’s discussion of equals(). To make sure you don’t forget this, use @Override just before the method.
This generates a warning, (visible in eclipse and probably netbeans, but not in Dr. Java) if the method does not in fact override anything.

 

Simple way to have good equals/hashCode: Just depend on Object’s equals and hashCode by not overriding these methods in your own element class.
Object equals just compares ref values, and Object’s hashCode hashes the ref value.

 

Ex: Take SimpleStudent, pg. 269(235), and drop equals and hashCode. The HashSet still works fine as long as you never create two different SimpleStudent objects for the same student. 
However, the main on that page does create two “Bob” objects, plus one “Joe”. With the Object equals, these are different elements, so the set will contain 3 elements.
If we use the equals that is coded there, the two “Bob” objects are .equals each other, and the set will contain only 2 elements. This is explained in the comments on lines 12-14. Both ways are fine for appropriate apps: you just need to understand how both work.

 

Need to fix up SimpleStudent, pg. 269(235):

int compareTo(SimpleStudent s)     <-- unlike equals, use the non-Object type here

Answer: No, it overrides equals, but not hashCode.  If it overrode neither, it would be OK, but once equals is overridden, you need to override hashCode too, and consistently.

The following example was not covered in class:

One more Set example: counting vowels.  We want something like:

 

  if (c is a vowel)

      count it

 

How do we code the condition here?  The ordinary programmer says

 

  if (c == ‘a’ || c == ‘e’ || c == ‘i’ || c == ‘o’ || c == ‘u’)

 

or maybe:

  String vowels = “aeiou”;

  if (vowels.indexOf(c) >= 0)  // search string for c

 

but with sets, we can avoid this search or repetitive comparison, once we build a Set of vowels:

 

    Set<Character> vowels = new HashSet<Character>();

  vowels.add(‘a’);  // Autoboxing here: char -> Character

  vowels.add(‘e’);

  vowels.add(‘i’);

  vowels.add(‘o’);

  vowels.add(‘u’);

 

in main loop over some string s:

  c = s.charAt(i)

  if (vowels.contains(c))   // assuming c a Character: O(1) lookup

     count++;

 

Q: Doesn’t this take a lot of time, searching for the character in the Set?

A: No, the JDK Sets are structured internally so that lookup for contains() is very fast.

Maps, Starting with language-neutral info

 

The picture here has a domain and range sets, with arrows from individual domain elements (of some type DomainType) to corresponding specific range elements (of type RangeType). 
Like a math function, each domain element has assoc. with it exactly one range element.  It’s OK for two arrows to land on the same range element,
but not OK for one domain element to have two arrows out of it.

 

In CS we think of the action of following the arrow most often as a "lookup" action.  For ex., employee records are looked up by social-security no. and/or by employee name. 
The first would need a map with ints for social security nos. as DomainType, Employee objects as RangeType.  The second, lookup by emp name, would need a map with a string type as DomainType, Employee objects as RangeType.

 

Let's look at a trivial example: reactions to grades (in class we did person to height):

 

'A'  ->   "Excellent"

'B'  ->    "Good"

'C'  ->    "OK"

 

Here the DomainType is char and the RangeType is string.  Each of these lines can be called a "key/value pair", or just “pair”. 
('A', "Excellent") is a pair of the grade 'A' (the key) and the phrase "Excellent" (the value).  The whole mapping is the set of these 3 pairs.

 

M = { ('A', "Excellent"), ('B', "Good"), ('C', "OK") }          --a map as a set of pairs, or “associations”

 

That's how a mapping turns out to be a collection, in the same league as the other collections we are studying, lists, stack, queues, and sets. However, in Java a Map has its own interface separate from Collection, in recognition of the importance of the domain-range view of Maps that is not captured completely by the “set of pairs” model. 
For one thing, not every collection of pairs makes a proper map: M qualifies as a map only if the collection of keys has no duplicates, i.e., constitutes a set.

 

Actions on Maps

We can add a key/value pair to a Map, and this operation is called put in Java (pg, 217), and we can lookup the associated range element (value) of any given domain element (key),
and this action is called get.  Put is more pushy than “set” for Lists – it can put another fact into the collection rather than just changing one that’s there.

 

Maps and arrays:

Now suppose we could do this:

         Arr['A'] = "yeah"

    Arr['B'] = "Good"

    Arr['C'] = "OK"

 

Note: Even in Java, chars are little numbers.  So ‘a’ + 1 is a good expression, and equals ‘b’.

 

This is meant to be a conceptual experiment, not a real C or Java setup. With this setup, we can look up the ‘A’ fact with array lookup.
Some languages allow arrays even with string indexes, Perl being one of them.  (When they are allowed, they are called "associative arrays".) 
The idea of maps is that we can have this effect, even if we can’t use array notation.

As holding conversions, like codes to grades, social security number to name

As generalized arrays

As math functions: y = f(x) is a map.

As a “database” with key lookup: SSno to employee record, ISBN to book record, name to inventory record.
